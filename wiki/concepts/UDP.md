# UDP

한 줄 요약: 신뢰성을 포기하고 속도를 택한 Transport Layer 프로토콜.

## 직관

UDP는 라디오 방송과 같다. 방송국은 신호를 계속 내보내고, 수신 여부를 확인하지 않는다. 누가 듣든 안 듣든, 잡음이 있든 없든 그냥 보낸다. 반면 TCP는 전화 통화 — 상대방 목소리를 들으며 연결을 확인한다.

## 특징

| 항목 | UDP | TCP |
|------|-----|-----|
| 연결 방식 | 비연결형 (connectionless) | 연결형 (3-way handshake) |
| 신뢰성 | 없음 | 있음 (ACK, 재전송) |
| 속도 | 빠름 (오버헤드 낮음) | 상대적으로 느림 |
| 순서 보장 | 없음 | 있음 |
| 헤더 크기 | **8 bytes** | 20 bytes 이상 |
| 흐름·혼잡 제어 | 없음 | 있음 |

## 사용 사례

신뢰성보다 **속도·실시간성**이 중요한 경우:
- 실시간 스트리밍 (영상, 음성)
- 온라인 게임
- DNS 질의 (빠른 응답이 핵심)
- VoIP (인터넷 전화)

## 세그먼트 구조

UDP의 데이터 단위 = **데이터그램(datagram)**

```
+------------------+------------------+
|   Source Port    | Destination Port |  16 bit each
+------------------+------------------+
|     Length       |    Checksum      |  16 bit each
+------------------------------------------+
|              Data (Payload)               |
+------------------------------------------+
```

**헤더 = 4개 필드 × 16bit = 8 bytes 고정**

| 필드 | 크기 | 설명 |
|------|------|------|
| Source Port | 16 bit | 송신 포트 |
| Destination Port | 16 bit | 수신 포트 |
| Length | 16 bit | 헤더 + 데이터 전체 길이 |
| Checksum | 16 bit | 오류 검출 (최소한의 무결성 보장) |

Payload 최대 크기: 65,507 bytes (이론값)

## UDP Checksum

UDP는 신뢰성이 없지만, **최소한의 오류 검출**은 제공한다.

### 계산 범위

단순히 UDP 데이터만이 아니라 **Pseudo Header**까지 포함:

| Pseudo Header 필드 | 설명 |
|-------------------|------|
| Source IP | 송신 IP 주소 |
| Destination IP | 수신 IP 주소 |
| Protocol | UDP 식별값 (17) |
| UDP Length | UDP 전체 길이 |

IP 계층 정보를 포함하는 이유: 잘못된 목적지로 전달된 패킷을 감지하기 위해.

### 계산 방식 — 1의 보수 덧셈

→ [[Ones-Complement-Sum]] 참고

1. Pseudo Header + UDP Header + Data를 16bit 단위로 나눔
2. 1의 보수 덧셈 수행 (carry는 하위 비트에 다시 더함)
3. 결과에 1의 보수(비트 반전) → Checksum

**송신**: Checksum 필드를 0으로 놓고 계산 → 결과를 삽입
**수신**: 동일 방식 재계산 → 결과가 0이면 정상, 아니면 폐기

### 주의

- IPv4: Checksum 선택 사항 (0이면 "사용 안 함")
- IPv6: 필수
- 오류 **검출**만 가능, 복구 기능 없음

## 왜 이렇게 설계됐는가

신뢰성 보장은 비용이 크다. TCP는 연결 설정(3-way handshake), ACK 대기, 재전송, 순서 관리를 모두 수행한다. 이 오버헤드가 실시간 서비스에서는 오히려 독이 된다.

UDP는 "신뢰성은 애플리케이션이 직접 처리하라"는 철학. DNS처럼 단순 요청-응답 구조는 패킷 하나 잃어도 그냥 다시 요청하면 된다. 스트리밍은 일부 패킷 손실이 화질 저하일 뿐, 멈추는 것보다 낫다.

Trade-off: UDP 위에서 신뢰성이 필요하면 애플리케이션 레이어에서 직접 구현해야 한다 (QUIC 프로토콜이 대표 사례).
