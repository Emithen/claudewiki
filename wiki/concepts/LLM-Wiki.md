# LLM Wiki

한 줄 요약: AI가 raw 소스를 읽어 직접 컴파일하고 유지하는 영구적 지식 네트워크.

## 직관

일반적인 AI 활용 방식은 매번 처음부터 답을 만들어낸다. 도서관에서 책을 빌릴 때마다 처음부터 읽는 것과 같다.

LLM Wiki는 다르다. 소스를 처음 읽을 때 AI가 핵심을 추출하고 위키로 컴파일해둔다. 다음번엔 이미 정리된 위키에서 꺼낸다. 지식이 쌓일수록 새 지식을 통합하는 비용이 낮아진다 — **복리 구조**.

```
기존 RAG: 소스 → [매번 검색 & 재발견] → 답변
LLM Wiki: 소스 → AI 컴파일 → 위키 ← AI가 꺼냄 → 답변
                              ↑
                         누적·연결됨
```

재귀적 성장과 구조적으로 동일: 기존 지식이 새 지식을 받아들이는 비용을 낮춘다.

## 개념 설명

### 3 레이어 구조

| 레이어 | 역할 | 소유자 |
|--------|------|--------|
| `raw/` | 불변 원본 소스 | 사람이 수집 |
| `wiki/` | 컴파일된 지식 네트워크 | AI가 작성·유지 |
| Schema (CLAUDE.md) | 운영 규칙·컨벤션 | 사람과 AI가 공동 설계 |

### RAG와의 차이

| | RAG | LLM Wiki |
|--|-----|----------|
| 지식 처리 시점 | 질문할 때마다 | 소스 추가할 때 한 번 |
| 인프라 | 벡터 DB, 임베딩 모델 필요 | 마크다운 파일만 |
| 지식 축적 | 없음 (매번 재발견) | 복리 (연결이 쌓임) |
| 신뢰도 | 문서 품질에 의존, 누락 가능 | 이미 정제된 위키에서 검색 |

### Gold In, Gold Out

"왜 수집했는지 말할 수 없으면 수집하지 않은 것과 같다."

목적 없는 수집은 노이즈. AI가 좋은 위키를 만들려면 입력 소스의 품질이 먼저다. Ingest 시 `## 인제스트 메모`에 이유를 남기는 것이 핵심.

### 워크플로우

```
수집 (Web Clipper) → raw/
          ↓
인제스트 (/ingest) → wiki/ 업데이트
          ↓
질문 (/query)     → wiki/ 기반 답변
          ↓
점검 (/lint)      → wiki/ 건강 유지
```

## 왜 이렇게 설계됐는가

Karpathy가 2026년 4월 제안. 기존 RAG의 두 가지 문제를 해결하기 위해:
1. **세팅 복잡도**: 벡터 DB, 임베딩 파이프라인 불필요
2. **지식 비축적**: 매번 처음부터 재발견하는 구조적 낭비

Trade-off: 위키 품질은 소스 품질과 스키마(CLAUDE.md) 설계에 달려 있음. 위키가 커질수록 index.md 기반 탐색의 신뢰도가 낮아질 수 있음 → 이 시점에 qmd 같은 로컬 검색 도구 고려.

Vannevar Bush의 Memex(1945) 개념과 유사 — 사적으로 큐레이션된 연관 링크 지식 저장소. Bush가 해결 못한 "유지보수를 누가 하나" 문제를 LLM이 해결.
