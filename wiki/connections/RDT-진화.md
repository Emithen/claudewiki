# RDT 진화 — 점진적 문제 해결 패턴

RDT(Reliable Data Transfer)의 1.0→3.0 진화는 "실패 시나리오를 하나씩 추가하며 최소한의 변경으로 해결"하는 패턴의 교과서적 사례.

## 진화 요약

| 버전 | 새로 추가된 가정 | 해결책 |
|------|----------------|--------|
| rdt1.0 | 완벽한 채널 | 그냥 송수신 |
| rdt2.0 | 비트 오류 가능 | ACK/NAK + 재전송 |
| rdt2.1 | ACK/NAK도 손상 가능 | Sequence Number |
| rdt2.2 | (NAK 제거) | ACK에 번호 부착 |
| rdt3.0 | 패킷 손실 가능 | Timeout + 재전송 |

## 각 버전의 핵심 통찰

**rdt2.0**: 오류를 피하는 게 아니라 **감지하고 알린다** (checksum + feedback)

**rdt2.1**: 피드백 자체도 손상될 수 있다 → **상태를 식별할 ID가 필요하다** (sequence number). 0, 1 두 개면 충분한 이유: stop-and-wait이라 한 번에 하나의 상태만 판정하면 됨.

**rdt2.2**: NAK는 사실 "직전 ACK를 다시 보내는 것"과 동일 → **ACK에 번호를 달면 NAK를 없앨 수 있다**

**rdt3.0**: 손실은 피드백이 아예 안 온다 → **"기다림에 기한을 두면" 감지 가능** (timeout)

## 구조적 유사성

이 패턴은 소프트웨어 설계에서 흔히 보인다:

- **Exception Handling 계층화**: 기본 동작 → 특정 예외 처리 → 더 구체적인 예외 처리
- **API 버전 업**: v1 → v2는 v1의 엣지 케이스를 처리, 기본 동작은 유지
- **방어적 프로그래밍**: happy path → 에러 처리 → 타임아웃 처리 → 부분 실패 처리

각 버전이 **이전 버전의 가정을 약화**시키며 더 현실적인 모델로 수렴하는 구조.

## 한계와 해결 — Pipelining

rdt3.0은 논리적으로 완전하지만 **Stop-and-Wait이라 효율이 낮다**.

```
효율 = (L/R) / (RTT + L/R)

RTT가 L/R보다 훨씬 크면 → 대부분의 시간을 기다리는 데 씀
```

Pipelining으로 해결: ACK를 기다리지 않고 여러 세그먼트를 연속 전송. 이는 TCP의 실제 구현 방식.
