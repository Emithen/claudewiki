# RAG vs LLM Wiki

두 가지 AI 지식 활용 방식의 구조적 차이.

## 한 줄 비교

- **RAG**: 질문할 때마다 원본에서 꺼낸다
- **LLM Wiki**: 미리 컴파일해둔 위키에서 꺼낸다

## 구조 비교

```
RAG
───────────────────────────────────────
질문 → 임베딩 → 벡터 DB 검색 → 청크 조각들 → LLM → 답변
                                    ↑
                              매번 원본에서 재발견


LLM Wiki
───────────────────────────────────────
질문 → index.md 탐색 → wiki 페이지 읽기 → LLM → 답변
                            ↑
                    이미 정리·연결된 지식
```

## 상세 비교

| 항목 | RAG | LLM Wiki |
|------|-----|----------|
| 지식 처리 시점 | 질문 시 실시간 | 인제스트 시 1회 |
| 인프라 | 벡터 DB + 임베딩 모델 | 마크다운 파일만 |
| 지식 연결 | 없음 (청크 단위 검색) | 있음 (위키 페이지 간 링크) |
| 지식 축적 | 없음 | 복리 (연결이 쌓일수록 강해짐) |
| 세팅 복잡도 | 높음 | 낮음 |
| 검색 신뢰도 | 문서 품질·누락 가능 | 정제된 위키 기반, 더 안정적 |
| 확장성 | 문서 수 무관하게 잘 작동 | 문서 많아지면 index 탐색 한계 |
| 유지보수 | 새 문서 → 재임베딩 필요 | 새 소스 → 인제스트로 통합 |

## 언제 무엇을 쓰나

- **RAG가 유리한 상황**: 수천~수만 개의 문서, 실시간 검색이 필요한 프로덕션 시스템, 문서 추가가 매우 빈번한 경우
- **LLM Wiki가 유리한 상황**: 개인 지식 관리, 소수의 큐레이션된 소스, 지식 간 연결과 복리 축적이 목적인 경우

## 구조적 유사성

RAG는 **stateless 함수** (입력 → 출력, 상태 없음)
LLM Wiki는 **stateful 데이터베이스** (지식이 누적·변화)

→ 연결된 개념: [[DataSource]] 에서 Connection Pool vs 매번 새 연결과 동일한 구조. 미리 만들어둔 연결(위키)을 재사용하는 것이 복리 효과를 만든다.
